# ADR 0004. Заводские PID: не сочиняем каталог, а даём задать пользователю

- Статус: принято
- Дата: 2026-07-14

## Контекст

Стандарт OBD-II (SAE J1979, Mode 01) описывает лишь общий набор параметров —
обороты, скорость, температура ОЖ, топливные коррекции и т.п. Всё, что реально нужно
для глубокой диагностики конкретного ЭБУ, лежит в **заводских модах** (обычно 21/22)
по адресам, которые производитель не публикует: у ВАЗ это Bosch M7.9.7 / M74 /
ME17.9.7 и «Январь 7.2», у остальных — свои.

Первоначально фича формулировалась как «добавить каталог mfr-PID для ВАЗ»: зашить
таблицу адресов и формул, чтобы приложение из коробки показывало заводские параметры.
Это и есть российский моат проекта, так что соблазн велик.

Проблема в том, что **проверяемого источника этих адресов у нас нет**. Они
расходятся по форумам и закрытым прошивкам сканеров, различаются между ревизиями
одного ЭБУ и не имеют официальной публикации. А ключевое свойство ошибки здесь —
асимметричное:

> Неверный адрес заводского PID возвращает **не ошибку, а правдоподобное неверное
> число.** ЭБУ отдаёт байты по запрошенному адресу; если адрес не тот, мы честно
> декодируем чужие байты и показываем, скажем, «угол опережения 23°», которого нет.

Обычный баг диагностируется по симптому. Этот — нет: он выглядит как рабочая функция
и ведёт владельца (и LLM-агента, который читает эти значения как факты) к неверному
ремонту. Для приложения, чья ценность — доверие к показаниям, это худший класс отказа.
Правило «не выдумывать коды и значения» в [AGENTS.md](../../AGENTS.md) существует
именно про это.

## Решение

**Каталога заводских PID в проекте нет. Пользователь задаёт их сам.**

Реализовано как расширяемый механизм, а не как данные:

- [`obd/CustomPid.kt`](../../app/src/main/java/ru/ngscanner/obd/CustomPid.kt) —
  описание PID: команда (`2101`, `22F190`…), смещение и число байт, множитель,
  сдвиг, знаковость, единицы. Декодирование чистое и покрыто тестами.
- [`ObdParser.dataBytesFor`](../../app/src/main/java/ru/ngscanner/obd/ObdParser.kt) —
  обобщение разбора ответа на произвольную моду (ответный заголовок = мода + `0x40`:
  `21` → `61`, `22` → `62`), а не только на Mode 01 (`41`).
- `Elm327.readCustom`, чтение в цикле опроса, плитка значений на панели приборов,
  форма в «Настройки → Свои PID».

В UI прямо сказано, почему адресов нет в комплекте и что их нужно брать из
проверенного описания (даташит ЭБУ, документация, собственные замеры).

Что сознательно **не** делаем:

- **Не зашиваем адреса «по памяти» LLM и не переносим их с форумов.** Ни агент, ни
  человек не пополняет каталог без проверяемого источника.
- Не поддерживаем произвольные формулы (только `raw × scale + offset`, 1–2 байта):
  выражение-парсер — это отдельный риск и его пока нечем оправдать.
- Не заводим «типовые неисправности по модели ВАЗ» как данные — это отдельная
  задача базы знаний, у неё свои требования к источникам (см. `docs/ecosystem.md`).

## Последствия

- Из коробки заводские параметры **не показываются** — это осознанный размен
  «меньше функций, но ни одного ложного показания». Пользователь с документацией
  получает полноценный доступ; пользователь без неё не получает вранья.
- Механизм провайдеро-независим: работает для любого ЭБУ (не только ВАЗ), потому что
  адрес и формула приходят снаружи, а не из захардкоженной таблицы.
- Появление **проверяемого** источника (официальный даташит, лицензионно чистая
  открытая база, собственные сверенные замеры на стенде) снимает запрет: тогда
  каталог заводится как предустановленные `CustomPid` с указанием происхождения —
  и это будет отдельный ADR с ссылкой на источник.
- Заводские PID сознательно вне графиков, поездок и «чёрного ящика»: они не входят
  в каталог `ObdPid`, и смешивать пользовательские данные с выверенными в тех
  подсистемах, где значения интерпретируются автоматически, преждевременно.
- Цена запроса: каждый свой PID — отдельная команда в цикле опроса, поэтому их число
  ограничено (`CustomPidRepository.MAX_PIDS`), иначе просядет частота обновления.
